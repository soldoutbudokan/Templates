# Dependencies

**Read this before making any change to TheTilt.** This file maps what feeds what so you (or Claude Code) can tell which downstream artifacts need a refresh — and, importantly, which ones don't refresh automatically and have hard-coded numbers in them.

---

## The two tiers

There are two kinds of artifact in the project:

1. **Auto-refreshed** — regenerated by `pipeline/run_pipeline.py` (or the data-only `refresh_tilt_data_only`). These never need manual editing for data changes; the GitHub Actions workflows handle them.
2. **Hand-written content with embedded numbers** — `.md` notes, the about page, the README, the leaderboard's hard-coded copy. These do **not** auto-update. When the model or data changes, the numbers in them go stale silently.

Almost every bug of "wait, why does the website say 1.54x but the analysis script says 1.61x?" comes from forgetting to refresh tier 2 after changing something in tier 1.

## Reverse map — "if you change X, refresh Y"

### If you change the **model** (`config/pipeline_config.yaml` hyperparams, `train_win_prob.py`, feature list, monotone constraints)

Run `python pipeline/run_pipeline.py` (full retrain), then check / refresh:

| Artifact | Why |
|:--|:--|
| `models/win_prob_lgbm.pkl` | Regenerated by the pipeline; commit it. |
| `public/data/*` | Regenerated by `export_json.py`; commit. |
| `public/about.md` §4 hyperparameter table | Pulls exact numbers from `pipeline_config.yaml`. |
| `public/about.md` §5 validation table (Brier / AUC / log loss / calibration) | These come out of the training run — copy from the new training-script output. |
| `public/about.md` §6 feature importance bar chart | Re-run `train_win_prob.py` and copy the new printed importances block. |
| `public/about.md` §11 top-10 table + "as of" date | Pull fresh top-10 from `tilt_rankings.json` and bump the date. |
| `public/about.md` §10 innings-asymmetry numbers | Re-run `notebooks/innings_bias_analysis.py` and copy the headline ratios + Spearman rho. |
| `public/notes/innings-bias.md` | Re-run `notebooks/innings_bias_analysis.py`. The notebook writes its own plot PNGs into `public/notes/plots/` and prints the headline numbers (mean WP shifts, ratios, top-N share %, rho). Refresh the prose tables from that printout. |
| `public/notes/kohli-2016-paradox.md` | Re-run `notebooks/kohli_2016_analysis.py`. The notebook writes 6 PNGs into `public/notes/plots/` and prints the per-match table (with `match_id`s for the in-table links), phase breakdown, dot-ball totals, and the comparison player-seasons. Refresh prose tables and any "Nx the TILT" headline phrases (the multipliers shift). |
| `public/notes/plots/innings_*.png` and `public/notes/plots/kohli_*.png` | Auto-regenerated by the two notebooks. The about-page copies (`about_*.png`) are static and don't auto-refresh. |
| `README.md` | Top-10 table, model metrics, feature-importance bars — same provenance as about.md §4–§6 / §11. |

### If you change **features** (`build_features.py`, the 15-feature list, shrinkage prior)

Same as a model change *and* check that:

- `public/about.md` §3 feature table is still correct.
- The shrunk-RR explanation in §3 still matches the constants in `build_features.py`.
- Any feature description in notes (e.g. `kohli-2016-paradox.md` references dot-ball penalties) still holds.

### If you change **data parsing / normalization** (`parse_matches.py`, venue map, team aliases)

- Match counts and ball counts in `meta.json` will change.
- `public/about.md` §2 (data section) and the headline coverage table at the top — refresh the numbers.
- `README.md` data section.
- Any blog post that quotes a match count.

### If you change **TILT computation / shrinkage** (`compute_tilt.py`)

- Every player's TILT changes — full leaderboard, per-player, GOATs all auto-refresh.
- `public/about.md` §8–§9 (computation, shrinkage table with k value, posterior CI math) — confirm the formulas and the k ≈ 5.3 number still hold.
- `public/notes/*.md` — every blog post implicitly assumes the current shrinkage k. If k changes materially the narrative may need re-checking.

### If you change **export thresholds** (`config.yaml` `min_matches`, `min_role_balls`)

- The `443 ranked players` figure in about.md and README will change.
- Refresh those.

### If you add a **new feature to the model**

- Update `public/about.md` §3 feature table and §6 feature-importance section.
- Update `README.md` feature table.
- Mention in any blog where it would change interpretation.

### If you change **website HTML/CSS** (no data implications)

- Nothing else needs refreshing. But re-test with `python3 -m http.server` from `public/` to confirm the page still renders.

## Hand-maintained content registry

Files containing hard-coded numbers, names, or claims that must be reviewed on every model retrain:

| File | What's in it that goes stale |
|:--|:--|
| `public/about.md` | Match/ball counts, hyperparameters, Brier/AUC/calibration, feature importances, top-10 table + "as of" date, k value, innings asymmetry ratios. |
| `public/notes/innings-bias.md` | Innings ball counts, mean WP shifts, the four phase ratios, top-N share %, Spearman rho, and the three innings-2-only feature importances. Source: `notebooks/innings_bias_analysis.py`. |
| `public/notes/kohli-2016-paradox.md` | Kohli 2016 vs 2019 headline table, full 2016 match log (with `match_id`s wired into in-table links), phase breakdown table, dot-ball table, comparison player-seasons table, runs-vs-TILT scatter `r`, GL credit-sharing numbers (ABD's TILT, Kohli WP at ABD's 100). Source: `notebooks/kohli_2016_analysis.py`. Headline multiplier phrases ("six times the TILT") drift with model retrains. |
| `README.md` | Match/ball counts, top-10 table, feature importances, model metrics. |
| `public/notes/plots/innings_*.png`, `kohli_*.png` | Regenerated by the two notebooks. The `about_*.png` siblings are static copies and don't auto-refresh. |

When publishing a blog post, **add a row here** with what numbers it embeds and which notebook produced them.

## What runs when

| Trigger | What it does | What it doesn't do |
|:--|:--|:--|
| `refresh-tilt-data.yml` (cron, twice daily) | Re-downloads Cricsheet, regenerates `public/data/*` using the committed pickle. Auto-commits if changed. | Doesn't retrain. Doesn't touch any `.md`. |
| `retrain-tilt-model.yml` (cron, March 1; manual dispatch) | Full pipeline including training. Auto-commits the new pickle and refreshed JSON. | Doesn't touch any `.md`. The hand-maintained content above must be updated by hand. |
| `python pipeline/run_pipeline.py` (local) | Same as the retrain workflow, locally. | Same caveat — doesn't touch `.md`. |
| `python -c "from pipeline.run_pipeline import refresh_tilt_data_only; refresh_tilt_data_only()"` | Same as the data-refresh workflow, locally. | Same caveat. |

## Workflow for any change to TheTilt

1. **Decide what tier you're touching.** Pipeline / model / data → tier 1. Anything in `public/*.md`, `public/*.html` text, or `README.md` → tier 2.
2. **If tier 1**, run the full pipeline locally (or trigger `retrain-tilt-model` workflow) so the JSON and pickle are consistent before commit. Then check this file's reverse map for which tier-2 files now need updating, and update them in the same PR/commit.
3. **If tier 2 only**, no pipeline run needed.
4. **Commit message convention** — if a model change forced refreshes to blogs/about/README, mention "and refreshed downstream content" in the message so future readers know why those files moved.
